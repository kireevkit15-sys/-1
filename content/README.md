# Content Pipeline -- RAG-пайплайн проекта РАЗУМ

Эта директория содержит весь контентный пайплайн: от сырых источников (транскрипты блогеров, академические статьи) до структурированных концептов и эмбеддингов для RAG-генерации вопросов.

---

## Структура директории

```
content/
  taxonomy.json              — индекс всех категорий (10 штук)
  README.md                  — этот файл

  categories/                — 10 JSON-файлов с полной таксономией
    decision-making.json
    emotional-intelligence.json
    financial-literacy.json
    leadership.json
    health-performance.json
    negotiation-relationships.json
    formal-logic.json
    critical-thinking.json
    rhetoric.json
    probability-statistics.json

  sources/
    bloggers/                — транскрипты блогеров (сырые и обработанные)
      markaryan/
        raw/                 — PDF/txt транскрипты (исходные файлы)
        processed/           — очищенные сегменты (JSON)
        concepts.json        — извлечённые концепты

    academic/                — академические источники (будущее)
```

### Категории

Файл `taxonomy.json` содержит мастер-индекс всех 10 категорий, разделённых на две ветки:

- **STRATEGY** (Стратегия) -- 5 категорий: принятие решений, эмоциональный интеллект, финансовая грамотность, лидерство, здоровье и производительность
- **LOGIC** (Логика) -- 5 категорий: переговоры и отношения, формальная логика, критическое мышление, риторика, вероятность и статистика

Каждый файл в `categories/` содержит полную таксономию категории: темы, подтемы, теги, уровни сложности.

## Пайплайн обработки контента

Контент проходит 6 этапов от сырого источника до готового вопроса:

```
PDF/txt  -->  Текст  -->  Сегменты  -->  Концепты  -->  Валидация  -->  Эмбеддинги  -->  RAG-вопросы
 Этап 1       Этап 1      Этап 2        Этап 3         Этап 4         Этап 5          Этап 6
```

### Этап 1: Парсинг (PDF -> текст -> очистка)

Входящие файлы (PDF, txt) преобразуются в чистый текст:

- **PDF-парсинг:** извлечение текста из PDF (pdf-parse / pdfjs-dist)
- **Очистка мусора:** удаление колонтитулов, номеров страниц, рекламных вставок, тайм-кодов
- **Нормализация:** исправление переносов, объединение разорванных абзацев, нормализация юникода
- **Выход:** чистый текстовый файл, готовый к сегментации

```
Вход:  sources/bloggers/markaryan/raw/lecture-01.pdf
Выход: промежуточный чистый текст для этапа 2
```

### Этап 2: Сегментация (разбивка на смысловые блоки)

Чистый текст разбивается на независимые смысловые блоки:

- **Размер сегмента:** 200-500 слов (оптимум для эмбеддинга и RAG)
- **Границы:** по смысловым переходам (смена темы, новый тезис, новый пример)
- **Перекрытие:** 10-15% overlap между соседними сегментами для контекста
- **Фильтрация:** сегменты без полезного контента (вода, повторы, реклама) помечаются `is_useful: false`

Результат сохраняется в `processed/` в формате segment JSON (см. ниже).

### Этап 3: Извлечение концептов (Claude Haiku -> structured JSON)

Каждый полезный сегмент обрабатывается Claude Haiku для извлечения структурированных концептов:

- **Модель:** Claude Haiku (дешёвая, быстрая, достаточная для извлечения)
- **Промпт:** системный промпт с примерами, schema-constrained output
- **Извлекается:** утверждение (claim), пример, теги, ветка (STRATEGY/LOGIC), уровень сложности
- **trust_level:** по умолчанию `needs_validation` -- ни один концепт не считается валидным без проверки
- **Батчинг:** до 10 сегментов за один запрос для экономии токенов

Результат сохраняется в `concepts.json` источника.

### Этап 4: Академическая валидация (PubMed API, cross-check)

Концепты со статусом `needs_validation` проходят автоматическую проверку:

- **PubMed API:** поиск по ключевым терминам концепта, проверка наличия подтверждающих исследований
- **Cross-check:** сопоставление claim с найденными abstract-ами через Claude Haiku
- **Результат валидации:**
  - `validated` -- найдено подтверждение в рецензируемых источниках
  - `opinion` -- утверждение субъективное, не поддаётся верификации (но может быть полезно)
  - `rejected` -- противоречит научным данным, не используется в генерации вопросов
- **academic_sources:** массив ссылок на PubMed/DOI для валидированных концептов

### Этап 5: Эмбеддинг (текст -> вектор -> pgvector)

Валидированные концепты и полезные сегменты преобразуются в векторные представления:

- **Модель эмбеддинга:** OpenAI text-embedding-3-small (1536 dims) или Cohere embed-multilingual-v3
- **Хранение:** PostgreSQL + pgvector расширение
- **Индексация:** IVFFlat или HNSW индекс для быстрого поиска
- **Метаданные:** source, branch, category, difficulty, trust_level, tags

### Этап 6: RAG при генерации вопросов

При генерации вопросов для батлов и разминки:

1. **Запрос:** категория + сложность + исключить уже заданные вопросы
2. **Поиск:** pgvector similarity search по релевантным концептам
3. **Контекст:** top-K концептов (K=3-5) передаются как контекст в Claude Sonnet
4. **Генерация:** Claude Sonnet генерирует вопрос с вариантами ответов на основе концептов
5. **Деdup:** проверка на дубликаты через cosine similarity с существующими вопросами
6. **Сохранение:** вопрос + метаданные + ссылки на source-концепты

```
Пользователь запрашивает батл "Принятие решений" (BRONZE)
  -> pgvector: найти 5 концептов по decision-making, difficulty=BRONZE
  -> Claude Sonnet: сгенерировать вопрос на основе концептов
  -> Dedup: проверить уникальность
  -> Сохранить в БД
```

## Форматы данных

### Concept JSON

Файл `concepts.json` содержит массив извлечённых концептов из источника:

```json
{
  "concept": "Правило 10-10-10 для принятия решений",
  "source": "markaryan/lecture-01",
  "claim": "При принятии сложного решения полезно оценить последствия через 10 минут, 10 месяцев и 10 лет — это снижает влияние эмоций",
  "example": "Человек решает уволиться из-за конфликта с начальником. Через 10 минут — облегчение. Через 10 месяцев — финансовые трудности. Через 10 лет — возможно, лучшая карьера.",
  "trust_level": "needs_validation | validated | opinion | rejected",
  "academic_sources": [
    {
      "title": "Temporal distancing reduces affective forecasting errors",
      "doi": "10.1037/emo0000319",
      "source": "PubMed"
    }
  ],
  "tags": ["decision-making", "emotional-regulation", "cognitive-bias"],
  "branch": "STRATEGY",
  "difficulty": "BRONZE"
}
```

**Поля:**

| Поле | Тип | Описание |
|------|-----|----------|
| `concept` | string | Краткое название концепта (1 строка) |
| `source` | string | Идентификатор источника (блогер/файл) |
| `claim` | string | Утверждение -- основной тезис, который можно проверить |
| `example` | string | Конкретный пример, иллюстрирующий концепт |
| `trust_level` | enum | Уровень доверия: `needs_validation`, `validated`, `opinion`, `rejected` |
| `academic_sources` | array | Ссылки на академические источники (PubMed, DOI) |
| `tags` | string[] | Теги для классификации и поиска |
| `branch` | enum | Ветка дерева знаний: `STRATEGY` или `LOGIC` |
| `difficulty` | enum | Уровень сложности: `BRONZE`, `SILVER`, `GOLD` |

### Processed Segment JSON

Файлы в `processed/` содержат массив сегментов:

```json
{
  "id": "markaryan-lecture01-seg-007",
  "source": "markaryan/lecture-01.pdf",
  "text": "Когда мы принимаем решение в состоянии стресса, активируется миндалевидное тело, которое запускает реакцию «бей или беги». Это приводит к тому, что мы выбираем краткосрочное облегчение вместо долгосрочной выгоды. Один из способов обойти это — метод 10-10-10: спросить себя, как я отнесусь к этому решению через 10 минут, 10 месяцев и 10 лет.",
  "topics": ["decision-making", "stress", "amygdala", "temporal-distancing"],
  "is_useful": true,
  "reason_if_rejected": null
}
```

**Поля:**

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | Уникальный ID сегмента (источник-файл-номер) |
| `source` | string | Путь к исходному файлу |
| `text` | string | Очищенный текст сегмента (200-500 слов) |
| `topics` | string[] | Автоматически определённые темы |
| `is_useful` | boolean | Содержит ли сегмент полезный контент |
| `reason_if_rejected` | string\|null | Причина отклонения, если `is_useful: false` |

## CLI-команды

Все команды запускаются из корня проекта через `pnpm`:

### Обработка сырого файла

```bash
pnpm content:process --source bloggers/markaryan/raw/lecture-01.pdf
```

Выполняет этапы 1-2: парсинг PDF, очистка, сегментация. Результат сохраняется в `content/sources/bloggers/markaryan/processed/`.

### Извлечение концептов

```bash
pnpm content:extract --source bloggers/markaryan
```

Выполняет этап 3: читает все processed-сегменты источника, извлекает концепты через Claude Haiku, сохраняет в `concepts.json`.

### Академическая валидация

```bash
pnpm content:validate
```

Выполняет этап 4: находит все концепты со статусом `needs_validation`, проверяет через PubMed API, обновляет `trust_level` и `academic_sources`.

### Генерация эмбеддингов

```bash
pnpm content:embed
```

Выполняет этап 5: берёт все валидированные концепты и полезные сегменты, генерирует эмбеддинги, сохраняет в pgvector.

### Генерация вопросов

```bash
pnpm content:generate --category decision-making --difficulty BRONZE --count 20
```

Выполняет этап 6: RAG-генерация вопросов по заданной категории и сложности. Параметры:

| Флаг | Описание | Пример |
|------|----------|--------|
| `--category` | Категория из таксономии | `decision-making`, `formal-logic` |
| `--difficulty` | Уровень сложности | `BRONZE`, `SILVER`, `GOLD` |
| `--count` | Количество вопросов | `20` (по умолчанию 10) |
| `--dry-run` | Только показать, не сохранять | (флаг без значения) |
| `--force` | Перезаписать существующие | (флаг без значения) |

## Добавление нового источника

Чтобы добавить нового блогера или источник:

1. Создать директорию: `content/sources/bloggers/<имя>/raw/`
2. Положить сырые файлы (PDF, txt) в `raw/`
3. Запустить обработку: `pnpm content:process --source bloggers/<имя>/raw/<файл>`
4. Извлечь концепты: `pnpm content:extract --source bloggers/<имя>`
5. Запустить валидацию: `pnpm content:validate`
6. Сгенерировать эмбеддинги: `pnpm content:embed`

Для академических источников -- аналогично, но в `content/sources/academic/`.

## Ограничения и лимиты

- **Claude Haiku:** используется только для извлечения концептов (этап 3) и cross-check (этап 4) -- дешёвые операции
- **Claude Sonnet:** используется только для генерации вопросов (этап 6) -- дорогая операция, лимит по количеству
- **PubMed API:** бесплатный, но rate limit 3 запроса/секунду без API-ключа, 10/сек с ключом
- **Эмбеддинги:** стоимость зависит от провайдера; OpenAI ~$0.02 за 1M токенов
- **pgvector:** требует PostgreSQL 15+ с расширением pgvector
